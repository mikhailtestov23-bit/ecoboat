# Ecoboat: инструкция по сборке проекта

Документ описывает стартовую сборку браузерной 3D-игры Ecoboat: локальный MVP, архитектуру игровых систем, лидерборд и план будущего деплоя.

## Текущий статус

Собран первый локальный MVP:

- 3D-озеро с берегами, препятствиями, мусором, зонами хищников и управляемым экокораблем.
- Раунд на 60 секунд, 3 жизни, очки за сбор мусора.
- Столкновения с препятствиями и атаки хищников снимают жизнь.
- Ненапряжная процедурная фоновая музыка и звуковые эффекты без внешних аудиофайлов.
- Четыре шторма за раунд с предупреждением за 3 секунды до опасной фазы.
- Локальный лидерборд через `localStorage`.
- Заготовка Supabase-лидерборда через переменные окружения.
- Vercel-ready production build через `npm run build`.

Локальный запуск:

```bash
npm install
npm run dev
```

Проверки:

```bash
npm run test
npm run build
```

Деплой:

- Vercel + Supabase - выбранный production-стек.
- Подробная инструкция находится в `DEPLOYMENT.md`.

## Короткое решение

Принятый стек:

- Vite + React + TypeScript - быстрый фронтенд-проект и удобная сборка статического приложения.
- Three.js через `@react-three/fiber` - 3D-сцена, камера, свет, вода, корабль и объекты.
- `@react-three/drei` - готовые помощники для камеры, загрузки моделей, окружения и текста.
- `@react-three/rapier` - физика и коллизии на базе Rapier, чтобы не писать столкновения вручную.
- Zustand или обычный React state - состояние раунда: время, жизни, очки, экран окончания игры.
- Supabase - внешний сервис для публичного лидерборда после локальной версии.
- Vercel - выбранный хостинг фронтенда и production-деплой.

Для первого локального прототипа внешний сервис не нужен: лидерборд можно хранить в `localStorage`. Для публичной версии используем Supabase, иначе таблица лидеров будет только на устройстве игрока.

## Цель MVP

Игрок управляет экологическим кораблем на ограниченном озере и за 60 секунд собирает как можно больше мусора. У игрока 3 жизни. Жизнь теряется при столкновении с препятствием или при атаке подводного хищника. После конца раунда игрок вводит имя, результат сохраняется в лидерборд.

Минимальный игровой цикл:

1. Стартовый экран с кнопкой начала игры.
2. Озеро с берегами, кораблем, мусором, препятствиями и зонами хищников.
3. Управление кораблем: вперед, назад, влево, вправо.
4. Счет, таймер 60 секунд, 3 жизни.
5. Сбор мусора увеличивает очки и убирает объект с поверхности.
6. Столкновение или атака снимает жизнь и дает короткую неуязвимость.
7. Конец раунда при `timeLeft === 0` или `lives === 0`.
8. Форма имени и запись результата.
9. Таблица лидеров.

## Игровая архитектура

### Озеро и границы

- Озеро делаем как плоскость воды в XZ-плоскости.
- Берега - визуальное кольцо вокруг воды плюс невидимые физические коллайдеры.
- Для MVP достаточно овального или многоугольного озера.
- Корабль не должен покидать воду: при касании границы он отталкивается или останавливается.

### Корабль

- Корабль - управляемое физическое тело.
- Движение ограничиваем по плоскости XZ.
- Управление:
  - `W` / стрелка вверх - вперед.
  - `S` / стрелка вниз - назад.
  - `A` / стрелка влево - поворот или движение влево.
  - `D` / стрелка вправо - поворот или движение вправо.
- Для приятного ощущения нужны ускорение, торможение, лимит скорости и плавный поворот.

### Мусор

- Мусор лежит или слегка покачивается на поверхности воды.
- Каждый объект имеет sensor collider.
- При контакте с кораблем:
  - объект исчезает;
  - начисляются очки;
  - может появиться новый объект в свободной точке озера.
- Типы мусора для MVP:
  - бутылка: 10 очков;
  - пакет: 15 очков;
  - канистра: 25 очков.

### Препятствия

Естественные препятствия:

- камни - статичные коллайдеры;
- маленькие островки - статичные коллайдеры и часть берега;
- бревна - кинематические объекты, которые дрейфуют по заданной траектории.

При столкновении корабль теряет 1 жизнь, получает короткую неуязвимость, а затем возвращается в безопасную точку или отталкивается от опасности.

### Подводные хищники

Хищник не преследует игрока по всему озеру. У каждого есть зона обитания:

- визуальный намек на зоне: тень под водой, круги на воде, плавник или пузырьки;
- sensor collider для периметра;
- если корабль заходит в зону, хищник атакует после короткого предупреждения;
- атака снимает 1 жизнь, если игрок не успел выйти из зоны.

Для MVP можно сделать 2-3 зоны с хищниками, которые двигаются по кругу внутри своей области.

### Раунд

Параметры:

```ts
export const ROUND_SECONDS = 60;
export const STARTING_LIVES = 3;
export const HIT_INVULNERABILITY_MS = 1500;
```

Состояния:

- `ready` - стартовый экран.
- `playing` - раунд идет.
- `paused` - пауза.
- `gameOver` - раунд завершен, показываем результат.
- `submittingScore` - отправляем результат.
- `leaderboard` - показываем таблицу.

## Лидерборд

### Локальный MVP

На первом этапе используем `localStorage`:

- `scores` хранится массивом объектов;
- после игры добавляем результат;
- сортируем по `score desc`, затем по `createdAt asc`;
- показываем топ-10.

### Публичная версия

Для публичного лидерборда нужен внешний сервис. Рекомендация: Supabase.

Причины:

- Postgres-таблица хорошо подходит для результатов.
- Supabase дает готовый API поверх базы.
- Можно включить Row Level Security.
- Можно добавить Edge Function для проверки и отправки результата, не раскрывая секретный ключ в браузере.

Минимальная таблица:

```sql
create table public.scores (
  id uuid primary key default gen_random_uuid(),
  player_name text not null check (char_length(player_name) between 1 and 24),
  score integer not null check (score >= 0 and score <= 100000),
  trash_collected integer not null check (trash_collected >= 0 and trash_collected <= 500),
  round_seconds integer not null default 60 check (round_seconds between 1 and 60),
  created_at timestamptz not null default now()
);

create index scores_rank_idx
  on public.scores (score desc, created_at asc);

alter table public.scores enable row level security;

grant select on public.scores to anon;

create policy "scores are publicly readable"
  on public.scores
  for select
  to anon
  using (true);
```

Для записи есть два варианта:

- Быстро для прототипа: разрешить `insert` через RLS с жесткими `check`-ограничениями.
- Лучше для публичной версии: писать результат через Supabase Edge Function или серверный endpoint на Vercel, где можно добавить rate limit, базовую проверку результата и антиспам.

Важно: если игра полностью считается в браузере, игрок технически может подделать результат. Для дружеского лидерборда достаточно ограничений и rate limit. Для честного соревновательного режима понадобится серверная валидация раунда или запись событий игры с проверкой.

## Деплой

### Выбранный вариант: Vercel + Supabase

Vercel хорошо подходит для Vite-приложения:

- фронтенд деплоится как статическая сборка;
- preview URL создаются для pull request;
- удобно хранить переменные окружения;
- позже можно добавить serverless endpoint, если не захотим использовать Supabase Edge Functions.

Настройки деплоя:

- Build command: `npm run build`
- Output directory: `dist`
- Environment variables:
  - `VITE_SUPABASE_URL`
  - `VITE_SUPABASE_PUBLISHABLE_KEY`
  - если будет серверная отправка: `SUPABASE_SECRET_KEY`

### Запасной вариант: Cloudflare Pages

Cloudflare Pages тоже подходит для Vite, но для Ecoboat не является основным направлением:

- Build command: `npm run build`
- Build output directory: `dist`
- После деплоя проект доступен на `<project>.pages.dev`
- Каждый push может автоматически пересобирать проект.

Если позже захотим уйти от Supabase, можно рассмотреть Cloudflare Workers + D1, но на старте Supabase проще и быстрее.

## Этапы разработки

### Этап 1: Играбельный прототип

- Создать Vite-проект.
- Сделать 3D-сцену озера, камеру и свет.
- Добавить корабль и управление.
- Ограничить движение берегами.
- Добавить мусор и сбор очков.
- Добавить таймер и жизни.
- Завершать раунд и показывать результат.
- Сохранять локальный лидерборд.

### Этап 2: Опасности и баланс

- Добавить камни, островки и бревна.
- Добавить зоны хищников с предупреждением атаки.
- Настроить потерю жизни, короткую неуязвимость и безопасный возврат.
- Настроить генерацию мусора так, чтобы объекты не появлялись в препятствиях.
- Провести баланс: скорость корабля, плотность мусора, опасность хищников.

### Этап 3: Публичный лидерборд

- Создать проект Supabase.
- Добавить таблицу `scores`.
- Подключить `@supabase/supabase-js`.
- Сначала читать топ-10 из Supabase.
- Затем добавить безопасную отправку результата через Edge Function или backend endpoint.
- Добавить ограничения имени игрока и проверку максимального результата.

### Этап 4: Полировка и деплой

- Добавить стартовый экран, паузу, экран конца игры.
- Добавить звуки: сбор мусора, удар, предупреждение хищника, конец раунда.
- Оптимизировать модели и текстуры.
- Проверить производительность на ноутбуке и мобильном экране.
- Подготовить Vercel-деплой.
- Проверить production build через `npm run build` и `npm run preview`.

## Проверки качества

Минимальные тесты:

- функция подсчета очков;
- окончание раунда по таймеру;
- окончание раунда при нуле жизней;
- сортировка лидерборда;
- валидация имени игрока.

E2E-проверки:

- игра запускается;
- корабль реагирует на управление;
- сбор мусора увеличивает счет;
- удар снимает жизнь;
- после конца раунда появляется форма имени;
- лидерборд показывает новый результат.

Визуальная проверка:

- на desktop и mobile canvas не пустой;
- корабль виден и не застревает у границ;
- HUD не перекрывает игровое поле;
- все тексты помещаются в свои элементы;
- камера не обрезает сцену при разных пропорциях экрана.

## Внешние сервисы

Обязательно позже:

- Supabase - база результатов лидерборда.
- Vercel - хостинг фронтенда.

Опционально после MVP:

- Sentry - ошибки в production.
- Web analytics - понять, играют ли люди и где бросают раунд.
- Cloudflare Turnstile - защита формы отправки результата, если появится спам.
- Хранилище ассетов - если 3D-модели и аудио станут тяжелыми. Для MVP можно хранить ассеты прямо в `public/`.
